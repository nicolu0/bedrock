MessagesHelper

How the demo agent makes Messages.app fire real typing dots, read receipts, and tapbacks.

The big picture

The demo agent needs to feel like a real person texting from a Mac. That means three things that AppleScript cannot do: turn the typing dots on and off, fire read receipts, and send tapbacks (heart, laugh, etc.). Those signals are emitted by Messages.app itself when it calls private Objective-C methods inside Apple's internal IMCore framework.

The only way to make those signals fire from the outside is to load our own code into Messages.app, so it runs with Messages.app's privileges and direct access to those private methods. That code is MessagesHelper.dylib, and the system that gets it talking back to our Node agent is what this doc describes.

The three pieces

FileRole
native/MessagesHelper/MessagesHelper.mObjective-C source. Compiled to a dylib. Runs inside Messages.app and calls private IMCore methods.
native/run-messages.shLaunches Messages.app with the dylib injected via DYLD_INSERT_LIBRARIES.
imessage-helper.mjsNode-side TCP server on 127.0.0.1:9772. The dylib dials in. Exposes a clean JS API (helper.markRead, helper.setTyping, helper.react, helper.ping) that demo_agent/imessage.mjs imports and calls.

So when you ask "what connects the demo agent to the native stuff?" — that's imessage-helper.mjs. It's the only file the agent code imports directly; everything below it (the socket, the dylib, the IMCore calls) is hidden behind that one module's API.

Two design choices that look weird but aren't

Why inject code instead of just using AppleScript?

AppleScript drives Messages.app from the outside via Apple Events. That's enough to send and read messages, but the typing indicator, read-receipt, and tapback methods aren't exposed to AppleScript. They live on private classes (IMChat, IMChatRegistry) that are only reachable from inside the Messages.app process. Hence the dylib.

Why does Messages.app dial out to us, instead of us calling in?

Messages.app runs under Apple's app sandbox. Its sandbox profile grants the network.client entitlement (outbound connect() is fine) but not network.server (inbound bind() / listen() is blocked). So an in-process server would never accept a connection.

The fix is to invert the IPC: our Node process is the server, listening on 127.0.0.1:9772, and the dylib inside Messages.app is the client, dialing out the moment it loads. Once the socket is up, both sides exchange newline-delimited JSON requests with monotonic IDs.

Wire format

One JSON object per line. Every request gets exactly one response with the same id.

// agent  -> helper
{"id":1,"op":"ping"}
// helper -> agent
{"id":1,"ok":true,"pid":12345}

// agent  -> helper
{"id":2,"op":"markRead","chatGuid":"iMessage;-;+15551234567"}
// helper -> agent
{"id":2,"ok":true}

// agent  -> helper
{"id":3,"op":"setTyping","chatGuid":"iMessage;-;+15551234567","typing":true}
// helper -> agent
{"id":3,"ok":true}

// agent  -> helper
{"id":4,"op":"react","chatGuid":"...","messageGuid":"...","reactionType":"love","summaryText":"hello"}
// helper -> agent
{"id":4,"ok":true}

If the agent restarts, the dylib reconnects with backoff. Any in-flight requests are resolved as {ok:false,error:"helper disconnected"} so callers never hang.

Why this needs a relaxed system config

macOS would normally refuse to inject an unsigned dylib into a hardened-runtime app like Messages.app. To get it to allow this on the Mac mini we did three one-time things:

  1. Disabled Library Validation system-wide so dyld won't reject our unsigned .dylib.
  2. Disabled SIP entirely from Recovery (csrutil disable) so the kernel allows DYLD_INSERT_LIBRARIES against a system app.
  3. Set the -arm64e_preview_abi boot-arg via nvram, so dyld will load arm64e third-party binaries (Messages.app on Apple Silicon is arm64e and won't accept an arm64-only library).
Heads-up: this configuration is permissive on purpose, scoped to this dedicated Mac mini. Don't replicate it on a daily-driver machine — disabling SIP and library validation removes some of macOS's strongest defenses against malicious code.

Day-to-day usage

One-time, after every code change to the dylib:

cd imessage/native/MessagesHelper && make

Each session you want the agent live:

# terminal 1: relaunch Messages.app with the helper injected
imessage/native/run-messages.sh

# terminal 2: start the agent — it listens on :9772 and waits for the
# helper to dial in (happens within ~1s of Messages launching)
node imessage/demo_agent/imessage.mjs

The agent will print helper online (Messages.app pid=…) once the dylib has connected and answered a ping. From that point on, every helper.markRead / setTyping / react call in the agent code shows up as a real signal on the recipient's phone.
