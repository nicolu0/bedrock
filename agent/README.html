agent/

An LLM-driven iMessage agent for property managers. One orchestrator, swappable skills, a shared tool catalog, and an iMessage transport. This document describes how the directory is laid out and what each piece does.

Layout

agent/ ├── server.mjs main entrypoint — chat.db poll + issue poll + transport ├── cli.mjs REPL testbed for the demo skill (no iMessage) ├── identity.mjs "who you are" prompt, prepended to every skill ├── memory.mjs per-handle profile + observations primitives ├── supabase.mjs tiny REST helper (env + fetchIssueById) ├── core/ │ └── orchestrator.mjs THE OpenAI tool-call loop. runTurn(skill, ctx). ├── skills/ per-trigger configs that drive the orchestrator │ ├── f1.mjs new work order → groupchat draft │ ├── chat.mjs PM reply in groupchat → tenant + vendor drafts │ └── demo.mjs 1:1 chat with a prospect ├── tools/ tool catalog, one file per tool │ ├── send_text.mjs │ ├── acknowledge.mjs │ ├── draft_tenant.mjs │ ├── draft_vendor.mjs │ ├── set_demo_stage.mjs │ ├── get_profile.mjs │ ├── update_profile.mjs │ ├── add_observation.mjs │ ├── recall.mjs │ ├── list_properties.mjs │ └── list_vendors.mjs ├── work-orders/ work-order plumbing (issue poller, workspaces, drafts UI, state) │ ├── issue-poller.mjs │ └── workspaces.mjs ├── imessage/ Messages.app bridge (helper.mjs + injected dylib) └── data/ per-handle memory: profile.json + observations.jsonl

Architecture

Three layers, with sharp seams:

  • Transport (server.mjs, imessage/, work-orders/issue-poller.mjs) — watches the world (chat.db, Supabase), decides which trigger fired, hands ctx to a skill, and turns the orchestrator's event stream back into iMessage signals or persisted drafts.
  • Skill (skills/*.mjs) — a config bundle: { name, model, maxIterations, tools, taskPrompt, buildContext, preCheck?, commit? }. The skill is what makes the agent behave differently for a new work order vs. a PM reply vs. a prospect chat.
  • Orchestrator (core/orchestrator.mjs) — the only place OpenAI is called. Loops over tool calls, dispatches inline, surfaces failures uniformly. Trigger-agnostic and skill-agnostic by design.

A skill tells the orchestrator how to assemble messages and what tools the model is allowed to call. The orchestrator never knows what a "demo" or an "f1" is — it just runs skill.buildContext(ctx), prepends the identity prompt + the skill's task prompt, streams the OpenAI completion, dispatches tool calls as their args parse, and returns { messages, drafts, toolCalls, failure }.

Grep-test invariant. core/orchestrator.mjs contains zero references to specific skill names (f1, chat, demo) or specific tool names (send_text, etc.). If you find yourself writing if (skill.name === ...) in there, the skill should own it via buildContext, a config field, or a tool result.

Orchestrator

runTurn(skill, ctx) is the entire surface. Both pollers and the CLI call it the same way; only ctx.sendMode and ctx.onEvent differ. 

// Skill shape
{
  name, model, maxIterations, maxTokens?,
  tools: [{ name, description, parameters, run(args, ctx) }, ...],
  taskPrompt: string | (ctx) => string,
  buildContext: (ctx) => OpenAI messages array,
  preCheck?:  (ctx) => result | null,    // short-circuit before LLM
  commit?:    (ctx) => void              // post-LLM hook
}

Loop

  1. preCheck. If the skill defines one and returns a result, short-circuit (no LLM call). Demo uses this for the canned opener on first contact.
  2. System prompt. identityPrompt + skill.taskPrompt (re-evaluated per iteration if it's a function — demo's stage block changes mid-turn).
  3. Stream. One streamed OpenAI chat-completion per iteration. As each tool call's args complete, the tool runs inline — the transport sees results as they happen instead of waiting for the whole response.
  4. Tool result conventions. Two generic fields the orchestrator honors without naming any tool:
    • assistantContent — concatenated into the assistant message between iterations so the model reads its own speech as plain text on the next loop.
    • endTurn — break the loop after this tool runs (used by set_demo_stage's closer).
  5. Failure surface. Tool throws, OpenAI errors, or maxIterations hit with no output → returned as { failure }. Pollers treat it as a warning and still ship any usable drafts.
  6. commit. Optional post-loop hook (demo persists the assistant's outbox to its in-memory conversation log).
  7. Telemetry. Every invocation appends one row to work-orders/state/turns.jsonl: skill, model, iterations, tool-call summary, drafts/outbox counts, failure, duration.

Send modes

The orchestrator passes ctx through to tools unchanged. Two values matter:

  • live — emit messages via ctx.onEvent({ type: 'message' }). The transport handles typing-dot dwell and dylib send. Used by demo and chat (acknowledge only).
  • draft — append messages to ctx.drafts for human review in the drafts UI. Used by F1 and the chat skill's tenant/vendor drafts.

Identity

identity.mjs exports one constant — identityPrompt — that's prepended to every skill's task prompt. Stable, small, never per-turn:

  • Who Bedrock is (the product, not a model).
  • Hard constraints that apply to every skill: no auto-send to customers, stay in character, don't invent facts, no em-dashes, surface tool failures honestly.

Skill-specific voice and stage logic live in each skill's taskPrompt, not here. Tool-specific behavior lives in each tool's description field — descriptions are part of the prompt and get the same engineering rigor as the task prompt itself.

Skills

A skill is a single object the orchestrator runs. The three live skills cover the three triggers we react to today.

f1new_issue

A new work order lands in Supabase (issues_v2). The issue poller waits for upstream agent_runs (intake + vendor) to be marked done, then runs F1 to produce a 1–2 message groupchat draft.

  • Model: gpt-4.1-mini by default. The prompt is a strict slot-fill template — no creativity, no fallback phrasings. Mini is plenty.
  • Send mode: always draft. The poller surfaces it in the drafts UI for the PM to Send.
  • Tools: send_text (one call per bubble).
  • Shape of output:
    • Message 1 (always): "Unit {unit} at {property} has {one-sentence summary}." — drops the "Unit X at" prefix if no unit; prepends URGENT: if urgent.
    • Message 2 (only if recommended vendor): "Should we send {vendor}?"
  • buildContext: formats ctx.issue into a bullet list of fields (property, unit, title, description, tenant, vendor, urgent flag). Tells the model "Recommended vendor: (none — skip Message 2)" explicitly when there's no vendor, so the model has clear signal to stop after one call.

chatgroupchat_reply

The PM replies in a mapped groupchat. The chat poller hands the message to this skill, which decides whether the reply is approving vendor dispatch on one of the recent F1 bundles we sent in that chat. If yes: live ack + tenant draft + vendor draft. If no: zero tool calls — the row is tagged no_match in the chat-log so a human can see it in the dashboard.

  • Model: gpt-5.4-2026-03-05. Correlation needs reasoning — the body text contains property, unit, and vendor names, and the model has to match those against the reply.
  • Send mode: live for the ack only (gated by the safety guard in send_text); the tenant and vendor outputs are draft by virtue of which tool produces them.
  • Tools: acknowledge, draft_tenant, draft_vendor.
  • Candidate set: last 5 groupchat sends within 7 days for this chat_guid, derived from sent-log.json by bundle_id. The model sees the actual message bodies plus their issue_id.
  • Decision rules (from the task prompt):
    • Clear yes on one issue → ack + draft_tenant + draft_vendor (in that order, same issue_id).
    • Vendor swap ("send Luigi instead") → no tools (v1 doesn't handle swaps).
    • Question, off-topic, or ambiguous across multiple candidates → no tools.

demoinbound_message

A prospect texts the agent's number from an unknown 1:1 handle. The demo skill walks them through a guided product example end-to-end — opener, setup (collect a property + plumber), dispatch (fake work order, get approval, "send" the vendor), learning (lock in the preference), followup (proactive nudge), complete (closer).

  • Model: gpt-5.4-2026-03-05. Voice work; the smaller model drifts.
  • Send mode: live. The transport drives typing dots, read receipts, and dylib sends.
  • Tools: send_text, set_demo_stage, plus the full memory surface (get_profile, update_profile, add_observation, recall, list_properties, list_vendors).
  • Stage machine: intro → setup → dispatch → learning → followup → complete, stored at system/stage in the per-handle profile. taskPrompt is a function that re-reads the stage every iteration so a mid-turn set_demo_stage applies on the next loop.
  • preCheck: on first contact, fires the canned opener (two lines) without an LLM call. Saves latency and rules out model variance on the entry message.
  • commit: appends the assistant's emitted bubbles to the in-process conversation log (per-handle Map; lost on restart for v1).
  • Voice guards: the prompt forbids em-dashes, opens-with-stock-acks, repeating questions verbatim across redirects, and ending turns on flat statements. Engagement-first, redirect-second.

Tools

Each tool is a single file exporting { name, description, parameters, run(args, ctx) }. The description is treated as part of the prompt — it carries behavioral rules the model needs at decision time. run reads anything it needs from ctx (handle, sendMode, onEvent, workspace_label, etc.) and returns a result object. Two reserved fields: assistantContent (text the orchestrator merges into the assistant message) and endTurn (break the loop).

send_textlivedraft

The only tool that emits a customer-visible message. One call per bubble — the prompt forbids newlines or stacked thoughts inside one call; the tool defensively splits on newlines anyway.

  • Live mode: emits ctx.onEvent({ type: 'message' }) per bubble. The transport handles typing-dot dwell and dylib send.
  • Draft mode: appends { body } to ctx.drafts. The poller writes a draft row to drafts.json.
  • Safety: in live mode, refuses if ctx.isPmHandle is true (belt-and-suspenders against ever live-sending to a PM number through the demo path).
  • Defensive drops: empty bodies, unfilled {placeholder} tokens, and the F1 empty-vendor leak "Should we send ?" are silently skipped.

acknowledgelive

Live ack into the current chat from the chat skill. Used before any draft tool, so the PM sees that we've received the message and are working on it.

  • Action space is enum-only: "got it", "on it", "okay", "thanks", "noted". The model cannot write anything off-script, which is what makes it safe to live-send into a customer-visible chat without trust + eval graduation.
  • Same emit shape as send_text live mode: ctx.onEvent({ type: 'message' }) + push to ctx.outbox.

draft_tenantdraft

Produces a tenant-facing draft for a given issue_id. Fetches the issue (Supabase), fills a hardcoded template using tenant + vendor names, writes a draft row with channel='tenant_appfolio'. The PM Copies it into AppFolio.

  • Errors gracefully if the issue has no tenant or no recommended vendor — returns { ok: false, error } instead of producing a half-template.
  • Tracks the new draft id on ctx.draftIds so the chat poller can tag the chat-log row with drafted vs no_match.

draft_vendordraft

Mirror of draft_tenant for the vendor side. Channel is vendor_appfolio; template uses the issue title (or description) in the "for X" slot. Same draft-id tracking, same Supabase fetch, same graceful failure when the recommended vendor is missing.

set_demo_stagelive

Advances the demo state machine by writing system/stage in the per-handle profile. The next iteration's taskPrompt picks up the new stage.

  • Closer special case: on the in-order followup → complete transition, the tool emits two hardcoded closing messages inline ("that's how we'll usually handle work orders.", "any questions?") and returns endTurn: true. Without that, the model would re-enter the new stage prompt and feel obligated to greet again.
  • Demo only. Other skills must not expose this tool.

get_profilememory

Reads a canonical fact by slug. Returns { value: string | null }. The prompt tells the model to call this before asking the user something we might already know.

update_profilememory

Writes or deletes a canonical fact about the current handle. Slugs are slash-paths:

  • user/<key> — e.g. user/name
  • property/<slug> — one entry per managed property
  • vendor/<trade>/<property-slug> — vendor name for a trade at a property
  • system/<key> — internal state (e.g. system/stage)

Empty value deletes the slug. Slugs are lowercase, dash-separated for multi-word values.

add_observationmemory

Appends a fuzzy note about the handle (preferences, anecdotes, context that doesn't fit a profile slug). Append-only, never edited. Optional tags help retrieval (["preference", "vendor"], etc.).

recallmemory

Searches past observations by query string. v1 ranking is keyword/substring overlap by hit count — cheap, good enough until embeddings make sense.

list_propertiesmemory

Returns every property this handle has mentioned managing, derived from profile slugs of the form property/<slug>. Zero-arg.

list_vendorsmemory

Returns vendors this handle has mentioned, optionally filtered by property and/or trade. Derived from vendor/<trade>/<property> slugs.

Transport

Transport turns the outside world into runTurn calls and turns the orchestrator's event stream back into iMessage signals or persisted drafts. Three pieces.

chat.db poller (server.mjs)

SQLite-readonly poll of ~/Library/Messages/chat.db every 1s. Pulls new iMessage rows that match either:

  • 1:1 direct messages from unknown handles → demo skill
  • Any chat whose GUID is in the workspace index (TEST_CHAT_GUID, JOSE_CHAT_GUID) → chat skill, but only when the sender is on that workspace's pm_handles list (tenant/owner replies in the same chat are dropped)

Per-handle (or per-chat) serialization via an inflight Map: a fast follow-up never races the in-flight turn, but different chats stay parallel. macOS Ventura+ stores message text in an NSAttributedString blob; extractText() lifts the UTF-8 string out. State (lastSeenRowId, dedup map) lives at agent/.imessage-state.json.

issue poller (work-orders/issue-poller.mjs)

Polls Supabase issues_v2 every 5s, scoped to the workspaces in workspaces.mjs. For each new issue:

  1. Readiness gate. Both upstream agent_runs for the issue (intake, vendor) must be done. Not-ready → pause the cursor and re-check next tick (the poller never skips an issue forward).
  2. Mark processed in the cursor file before generating, so a mid-draft crash doesn't double-fire.
  3. Run F1 with sendMode='draft', isPmHandle=true (belt-and-suspenders for the send_text safety guard).
  4. Persist draft. Resolves participants from chat.db, writes one draft row carrying messages: [{ body }, ...]. Weekend-hold (hold_until = next Mon 09:00 PT) applied unless the issue is urgent.

F1 failures are logged as warnings, not skipped — if the loop produced a usable bundle (drafts present), the human still sees it.

iMessage bridge (imessage/)

Messages.app is sandboxed and can't bind a TCP listener, but its network.client entitlement allows outbound connect. So the topology is inverted: the JS process listens on 127.0.0.1:9772; the dylib injected into Messages.app dials out. Once connected, both sides exchange newline-delimited JSON requests with monotonic ids.

helper.mjs exposes the surface, never throws — every call resolves to { ok, ... } or { ok: false, error }: 

helper.ping()
helper.markRead(chatGuid)
helper.setTyping(chatGuid, typing)
helper.react(chatGuid, messageGuid, reactionType, summaryText)
helper.send(chatGuid, text)

server.mjs maps orchestrator events to dylib calls per incoming message:

  • read → markRead(chatGuid) (also fired immediately on every inbound, before the agent runs)
  • typing → intentional no-op (see callout)
  • message → setTyping(true), dwell, setTyping(false), send
  • tool_call → log only

Typing dwell scales with message length (min 300ms, +25ms/char, max 2000ms) so a four-word reply doesn't feel as instant as a thirty-word one.

Why typing dots are tied to messages, not typing events. The orchestrator emits a typing event at the start of every iteration including the final no-op one where the model returns no more tool calls. Reacting to those events directly causes a ghost flash of dots after the last message. Dots come on only immediately before a real message ships — mirroring real iMessage behavior.

Host setup (Library Validation off, SIP off, -arm64e_preview_abi boot-arg) is documented in imessage/README.md. This is a dedicated Mac mini — do not replicate on a daily-driver.

Running it

# terminal 1: launch Messages.app with the dylib injected
agent/imessage/run-messages.sh

# terminal 2: start the agent server (chat poll + issue poll + drafts UI + transport)
node agent/server.mjs

# REPL testbed for the demo skill, no iMessage in the loop
node agent/cli.mjs +15555550100

Required env: OPENAI_API_KEY, plus the Supabase + workspace vars listed in the project root CLAUDE.md. Optional: DEMO_HANDLE scopes server.mjs to a single 1:1 handle for safe testing; HELPER_DISABLED=1 skips dylib calls (no typing dots / read receipts) for orchestrator-only debugging; WORK_ORDERS_REQUIRE_READY=0 bypasses the F1 readiness gate.
