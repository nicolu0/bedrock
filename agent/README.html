agent/

An LLM-driven iMessage agent for property managers. This document describes how the directory is laid out and how the pieces fit together.

Layout

agent/ ├── server.mjs main entry — iMessage transport + poll loop ├── cli.mjs REPL testbed (no iMessage) ├── core/ model-driven turn logic, independent of transport │ ├── orchestrator.mjs │ ├── prompts.mjs │ ├── tools.mjs │ └── memory.mjs ├── data/ per-handle profile.json + observations.jsonl ├── imessage/ Messages.app bridge library │ ├── helper.mjs │ ├── MessagesHelper/ .dylib injected into Messages.app │ ├── run-messages.sh │ ├── docs.html │ └── README.md ├── dev_tools/ manual / developer-only utilities │ ├── issues_poller.mjs │ └── inject-draft.applescript └── docs/ design docs (SPEC, prompt notes) ├── SPEC.html └── prompt_refactor.html

The two layers

The directory splits cleanly along one seam: transport vs. agent logic. Everything in core/ is pure — it takes a handle and a user message and emits an event stream. Everything in imessage/ + server.mjs deals with the Messages.app side: watching chat.db, sending replies, driving the typing dots and read receipts.

Because the seam is clean, cli.mjs can run the same orchestrator with no iMessage in the loop — it just prints events to the terminal. Useful for iterating on prompts without spamming a real phone.

Entry points

CommandWhat it does
node agent/server.mjs The product. Polls chat.db for unread messages from unknown numbers, runs a turn through the orchestrator, and ships replies via the dylib bridge. Filters out known customer/vendor handles so testing doesn't hijack real conversations.
agent/imessage/run-messages.sh Launches Messages.app with MessagesHelper.dylib injected. Must be running before server.mjs — the helper dials out to the JS-side TCP listener on port 9772.
node agent/cli.mjs [handle] REPL testbed. Same orchestrator, no Messages.app. Has :state, :reset, :replay commands.
node agent/dev_tools/issues_poller.mjs Manual workflow: polls Supabase for new tenant work-order requests and injects drafts via the clipboard. Separate from the agent until drafting + sending is wired into core/.

How a message flows through

  1. chat.db poll. server.mjs queries the SQLite database every 1s for new iMessage rows in 1:1 chats (no group chats, no attachments). macOS Ventura+ stores message text in an NSAttributedString blob — extractText() lifts the UTF-8 string out.
  2. Filter. Drop messages from known handles (real customers/vendors), drop messages already processed (state.processed in .imessage-state.json), drop self-sent messages.
  3. Per-handle serialization. An inflight map chains turns per handle, so a fast follow-up doesn't race the in-flight turn. Different chats stay parallel.
  4. Orchestrator. runTurn(handle, text, { onEvent, ctx }) appends to in-memory conversation history, builds the system prompt from the current system/stage (intro → setup → dispatch → learning → followup → complete), and streams the OpenAI chat completion.
  5. Tool dispatch. Tool calls are dispatched inline as the stream produces them — each call's args parse, the tool runs, and the result feeds back into the next iteration. send_text calls become message events; everything else becomes tool_call events.
  6. Transport mapping. server.mjs's onEvent handler maps events to dylib calls: read → markRead; message → setTyping(true), dwell, setTyping(false), send. Typing dwell scales with message length so a short reply doesn't feel as instant as a long one.
Why typing dots are tied to messages, not typing events. The orchestrator emits a typing event at the start of every model iteration, including the final no-op one where the model returns no more tool calls. Reacting to those events directly causes a ghost flash of dots after the last message. Dots come on only immediately before a real message ships — mirroring how iMessage works in real life.

core/ — the agent

FileRole
orchestrator.mjsStreaming tool-call loop. One runTurn per user message. Re-reads the stage at every iteration so a mid-turn set_demo_stage applies on the next loop. Has a hardcoded opener (no model call on first turn) and a hardcoded closer for the followup → complete transition.
prompts.mjsPer-stage system prompts plus the canned OPENER_MESSAGES and FOLLOWUP_COMPLETE_CLOSER lines.
tools.mjsOpenAI function-calling tool definitions and their implementations: send_text, update_profile, get_profile, add_observation, recall, list_properties, list_vendors, react_to_message, set_demo_stage.
memory.mjsPer-handle JSON-file storage. profile.json is a flat slug → value map (slugs like user/name, property/<slug>/units, vendor/<trade>/<property-slug>). observations.jsonl is append-only fuzzy notes with naive substring-overlap recall.

Memory model

Two slots, deliberately. Profile is for facts that belong on a key — a property's unit count, a vendor's name for a trade. The slug shape (vendor/plumbing/oak-st) makes listProperties / listVendors trivial prefix scans. Observations are fuzzy — preferences, anecdotes, context that doesn't fit a slug. Recall is keyword-substring with hit-count ranking, not embeddings; cheap and good enough for now.

Demo state machine

A new handle starts at the intro stage. The model advances by calling set_demo_stage: setup → dispatch → learning → followup → complete. Each stage has its own system prompt block. The orchestrator refreshes the prompt every loop iteration, so the stage can change mid-turn. The followup → complete transition is the one place we override the model: a hardcoded closer ships and the turn ends, so the model never enters the complete-stage prompt mid-turn and feels obligated to greet again.

imessage/ — the bridge

Messages.app is sandboxed and can't bind a TCP listener, but its network.client entitlement allows outbound connect. So the topology is inverted from what you'd expect: the JS process listens on 127.0.0.1:9772; the dylib inside Messages.app dials out. Once connected, both sides exchange newline-delimited JSON requests with monotonic ids.

helper.mjs exposes the surface: 

helper.ping()
helper.markRead(chatGuid)
helper.setTyping(chatGuid, typing)
helper.react(chatGuid, messageGuid, reactionType, summaryText)
helper.send(chatGuid, text)

Every call resolves to { ok, ... } or { ok: false, error } — never throws. send replaced an earlier osascript-shelling approach so all four operations now share one channel. See imessage/docs.html for the dylib internals.

dev_tools/ — manual utilities

issues_poller.mjs polls Supabase for new tenant work-order requests and injects drafts into Messages.app via clipboard paste (inject-draft.applescript). It is run by hand, not part of the product. When the agent itself can draft and send work-order replies, the useful parts (Supabase polling) lift into core/ or server.mjs; the rest gets retired.

Running it

# terminal 1: launch Messages.app with the dylib injected
agent/imessage/run-messages.sh

# terminal 2: start the agent server
node agent/server.mjs

# or, no iMessage:
node agent/cli.mjs +15555550100

Requires OPENAI_API_KEY in .env (repo root or agent/.env). Optional DEMO_HANDLE scopes the server to a single handle for safe testing; HELPER_DISABLED=1 skips the dylib calls (no typing dots / read receipts) for orchestrator-only debugging.
