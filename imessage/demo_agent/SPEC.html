C. hardcoded bans (negative pinning — what the model must NOT say)

  • stock acks prompts.mjs:50 — "got it", "okay", "sounds good", "great", "perfect", "thanks", "thanks for letting me know", "moving on", "moving over to a live one"
  • echo confirmations prompts.mjs:50 — "got it, mariposa"
  • meta-narration of next message prompts.mjs:156 — "here's what i'd text you", "i'd text you exactly like this", "this is what i would say"
  • pre-narrating the flow prompts.mjs:42 — "next i'll show you...", "now i'll demonstrate..."
  • action-anticipating verbs prompts.mjs:163–164 — "sending mario", "dispatching mario"
  • complete-stage greeting regress prompts.mjs:235 — "what can i help with" / re-introductions after the demo wraps
  • the word "default" in learning turn 1 prompts.mjs:192
  • llm/ai/model/assistant talk prompts.mjs:39, 254

D. stage-driven beats — pinned in structure, not words

The number of messages and what each one is for are pinned, but wording is freely generated.

  • setup turn 1 — 2 messages (framing + property question) prompts.mjs:132–136
  • dispatch entry — 2 messages (pms-context framing + the actual work order) prompts.mjs:155–158
  • dispatch approval — 2 messages (decision + completed action) prompts.mjs:162–164
  • followup turn 1 — 3 messages ("also keep an eye" + "fast forward" + "gone quiet, follow up?") prompts.mjs:212–216
  • followup turn 2 — 3 messages (response + B1 msg 2 + B1 msg 3) prompts.mjs:222–228
  • complete-stage capability explanations — 2–3 short messages, never one packed list prompts.mjs:239–248

tightening levers, ranked by ROI

  1. Promote B1 into truly-hardcoded. The closer is already prescribed VERBATIM and the model has zero creative latitude. Move into the orchestrator: when stage advances to complete, push the two strings directly into the outbox. No model call.
  2. Promote A1's "yes" branch. When the user replies "yes/sure/ok/yeah" to the opener, the intro stage just advances and asks for a property. The framing message + property question could be pinned (or at least the property question).
  3. Decide on B2. If you want it predictable, lift to VERBATIM and hardcode. If you want variability, leave it.
  4. B5 first-name response is a candidate for "code path, not prompt", but harder to factor cleanly because the model still needs to redirect back to whatever beat we were on.

part 2 — prompts.mjs merge & prune plan

context

imessage/demo_agent/core/prompts.mjs has grown by accretion — every observed failure added a new rule or example. Several rules are now stated 2–4× across BASE and the per-stage blocks, one BASE example actively contradicts a newer dispatch rule, and a handful of examples illustrate rules already covered by other (more vivid) examples. This pass consolidates duplicates so the model gets a tighter, less self-conflicting prompt and so future edits have a single canonical place to land.

This is purely a refactor of the prompt text. No behavior change is intended; the goal is parity with current behavior at lower token count and zero internal contradictions.

critical file

imessage/demo_agent/core/prompts.mjs (only file modified)

findings

A. stock-acknowledgment rule stated twice

  • BASE line 50: don't open replies with stock acknowledgments ("got it", "okay", "sounds good", "great", "thanks for letting me know")...
  • Dispatch line 172: do NOT open with stock acknowledgments ("got it", "great", "perfect", "thanks", "moving on", "moving over to a live one", etc.)

Action: Merge banned-word lists into the BASE rule. Drop the dispatch repeat. Keep "moving over to a live one" / "moving on" by folding them into the BASE list.

B. "if they go off-topic, answer briefly then bring it back" stated 4×

setup line 164, dispatch line 195, learning line 232, followup line 259

Action: State once in BASE (# how you write), drop from each stage block.

C. "don't describe the demo / break character" — overlapping and partially self-contradictory

  • BASE line 39 — no meta about being a demo/llm/ai
  • BASE line 42 — never describe the demo or the flow in the abstract. never say "say a work order comes in"
  • Dispatch line 197 — never describe the demo or list the flow. just be in the work order.
  • Complete line 284 — never break character. never explain that this was a demo.

Tension: the dispatch transition explicitly frames the scenario as an example ("let's say a tenant at <property> just submitted a maintenance request"). So "never explain that this was a demo" in complete contradicts our own behavior — we ARE acknowledging it's an example.

What we actually want:

  1. Bedrock is the product, not an LLM/AI/assistant. Don't talk about model internals or being an AI. (BASE line 39 — keep.)
  2. Don't meta-narrate the flow ahead of time ("next i'll show you X then Y"). The demo should feel like a real conversation, not a guided tour. (BASE line 42's intent — keep, reword.)
  3. Acknowledging that something is an example or hypothetical scenario is fine and explicitly encouraged at the dispatch transition.

Action:

  • Reword BASE line 42: "don't narrate the flow in advance ('next i'll show you...', 'now i'll demonstrate...'). framing a scenario as 'let's say' or 'imagine' is fine — that's how a real walk-through reads."
  • Drop dispatch line 197 (subsumed by reworded BASE).
  • In complete, REMOVE "never explain that this was a demo". Keep "never break character" but clarify: "stay in character as bedrock the product (not an llm/ai/assistant). the user knows this was a walk-through; that's fine. what's not fine is suddenly talking about being a model."

D. "one thought per send_text / multi-message turns expected" stated 2×

  • BASE line 48 — one thought per message... never put a line break inside a single send_text
  • Followup line 257 — multi-message turns are expected here. one thought per send_text. no line breaks inside a single message.

Action: Drop the followup duplicate.

E. "don't echo property name back as confirmation" duplicates BASE

  • Setup line 155 — reminds the model not to echo the property name
  • BASE line 50 — already says don't echo their words back as confirmation ("got it, mariposa")

Action: Drop the explicit reminder in setup.

F. generic "bad replies" example block (lines 100–114) is redundant

That block re-demonstrates the no-stock-acks / no-echoing rule already covered by the rule itself (line 50, "got it, mariposa"), the "i remember the useful stuff" good/bad pair (lines 77–85), and the "what's my name" example (lines 87–96).

Action: Drop lines 100–114. Three good/bad pairs for the same rule cluster is overkill.

G. CONFLICT: BASE example contradicts new dispatch rule

  • BASE line 41 — write it as if it's already happening ("on it. sending mario, tenant's been notified")
  • Dispatch line 181 (recently added) — "sending mario" anticipates the action — use "i texted mario" instead.

Action: Fix BASE example to "on it. i texted mario, tenant's been notified" so BASE and dispatch agree.

H. learning-stage flow is structurally awkward

Case A and Case B are introduced before "turn 1", but Case B's handling exits the stage entirely (jumps to followup) while Case A continues into "turn 2". The current ordering — Case A → Case B → turn 2 — makes turn 2 read as if it might apply to Case B too, which it doesn't.

Action: Restructure so Case B is an early short-circuit (if a prior observation already exists, do X and skip to followup), then the rest reads linearly as the normal Case A flow.

out of scope for this pass

  • Stage isolation, eval loop, fine-tune — items 3, 4, 5 in the next batch.
